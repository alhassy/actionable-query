* Elevator pitch

Emacs is /full/ of interfaces built on the pattern "query some data
source; show the results; let the user act on them." Every Emacs user
already lives inside one: =M-x= itself is a query over all interactive
commands, with ~RET~ to execute.

Same pattern, different substrate, repeated all over Emacs:

| Command        | Query                                | Actions                                 |
|----------------+--------------------------------------+-----------------------------------------|
| =M-x=            | all interactive commands             | ~RET~ to execute                          |
| =dired=          | files in a directory                 | visit, rename, delete, chmod, shell out |
| =proced=         | running processes                    | signal, kill, renice                    |
| =buffer-menu=    | open buffers                         | switch, kill, save                      |
| =org-agenda=     | TODOs and scheduled items            | reschedule, mark done, drill in         |
| =magit-log=      | git commits                          | show, checkout, revert                  |

=actionable-query= extends the tradition to *arbitrary* data sources.
The query can be a shell command, an HTTP request, an RSS feed, a
bookmark list, a curated personal playlist, a JSON payload — anything
that produces a list of items. The actions are whatever you declare them
to be.

Under the hood we use =vtable= for tabular rendering and =org-ql-views=
for view registration — so =org-ql-view= opens your views by name, =g=
refreshes, and =q= buries the buffer, all with zero extra wiring.

** Why not just =vtable=?

=vtable= renders a table.  It doesn't answer:

- *How do I fetch data asynchronously without freezing Emacs?*
  ~url-retrieve~ is callback-driven; ~make-vtable~ expects data up-front.
  Wiring a spinner, swapping in rows on callback, and handling ~g~-refresh
  is non-trivial boilerplate every async view duplicates.

- *How do I let users snooze/dismiss rows and have that survive a session restart?*
  =vtable= has no persistence layer.  You'd need a hash table, a date-based
  expiry scheme, =savehist= registration, and resurrection logic — per view.

- *How do I surface all my keybindings discoverably?*
  =vtable='s ~:actions~ silently binds keys; users who don't read the source
  won't find them.  A ~?~ transient that lists every action with its
  description is a separate =transient= integration to build from scratch.

- *How do I group results from multiple async sources into one buffer?*
  Fetching N feeds in parallel, assembling them into titled sections, and
  rendering each as a separate vtable — with a unified snooze/filter layer
  on top — is substantial orchestration that has nothing to do with the
  domain logic of any single view.

- *How do I wire a row to an Org heading so standard =org-agenda= nav
  (RET, TAB, I, o) just works?*
  The ~org-marker~ / ~org-hd-marker~ text-property protocol, kept live on
  every cursor movement, is an ~org-agenda-mode~ implementation detail most
  users shouldn't have to know.

- *How do I get clock-in, scheduling, TODO cycling, and all the other
  org-agenda commands working on arbitrary rows?*
  ~agenda.el~ closes that gap: any view with ~:org-upsert~ wired up gets the full
  suite of org-agenda actions automatically — including auto-creating an Org
  heading in ~org-default-notes-file~ when none exists yet.

=actionable-query= is the ergonomic layer on top of =vtable= that answers
all of these — the same way =dired= is to raw file I/O and =proced= is to
raw process listings.  One macro, ~actionable-query-defview~, and you get
async loading, session-persistent snoozing, a ~?~ transient, grouped
multi-table rendering, column filters, bulk marking, row reordering,
hearting, auto-refresh, RSS/Atom/CLI helpers, Org-marker wiring, full
org-agenda command parity, and org-ql query views — all without touching
any of the underlying machinery yourself.

* Installation

No MELPA yet. Clone the repo and load from your init:

#+begin_src emacs-lisp
(load-file "~/actionable-query/actionable-query.el")
#+end_src

Dependencies: =org-ql= (MELPA), =vtable= (built into Emacs 29+),
=transient= (built into Emacs 29+).

~agenda.el~ and ~actionable-query-scheduler.el~ are loaded automatically
by the main file.

** Startup behavior: views registered silently, opened on demand

Evaluating an ~actionable-query-defview~ form does two things:

1. *Registers* the view in ~org-ql-views~ (so ~M-x org-ql-view~ finds it).
2. *Opens* the view buffer ---but only when evaluated interactively.

The key variable is ~aq-defview-open-on-define~, whose default is
~'auto~.  Under ~'auto~, the library checks ~load-file-name~ at expansion
time:

- ~nil~ ---form evaluated interactively via ~C-x C-e~, ~ielm~, or ~M-:~---
  → buffer opens immediately.  This is the "see what you just
  defined" ergonomic that makes iterative development fast.

- Non-nil ---file being loaded via ~load-file~, ~require~, or
  ~org-babel-load-file~--- → view is registered silently.  No
  unsolicited buffers on startup.

You never need to set this variable yourself.  The behavior is
correct by default: define a view and press ~C-x C-e~ to see it; load
a file of views at startup and they are quietly available via
~M-x org-ql-view~ or their named command.

*** If you want the old "always open" behavior

Set the variable to ~t~ before loading your view files:

#+begin_src emacs-lisp
(setq aq-defview-open-on-define t)
(load-file "~/my-views.el")
#+end_src

This is equivalent to the pre-~'auto~ behavior and is useful if you
actively want Emacs to open a specific dashboard on every startup
---say, a Morning Standup view that you read the moment Emacs loads.

*** If you use ~desktop-save-mode~

~desktop-save-mode~ can restore ~*Org QL View*~ buffers from a
previous session.  Since each view's ~:objects~ thunk re-fires on
open (via ~g~ or by calling the view function), this means your
queries run automatically on every Emacs restart ---which may or may
not be desirable.

To opt out, add the ~*Org QL View*~ buffer name pattern to
~desktop-clear-preserve-buffers~ or exclude it from restoration:

#+begin_src emacs-lisp
(add-to-list 'desktop-clear-preserve-buffers-regexp
             (regexp-quote "*Org QL View"))
#+end_src

To opt in deliberately (run all your queries fresh on restart), just
leave ~desktop-save-mode~ as-is; the view buffers will be restored
and their data refreshed.

* Hello, world

Maybe start with super simple:

(actionable-query-defview :objects '("one" "two" "three"))

Then add actions?

The smallest useful view: three strings, each with a message on ~RET~.

#+begin_src emacs-lisp
(actionable-query-defview my/first-view
  "My first view"
  :objects '("one" "two" "three")
  :actions '(("RET" "Print it" (lambda (it) (message "You picked: %s" it))))
  :help-echo (lambda (it) (format "Row value: %s" it)))
#+end_src

Open it with ~M-x org-ql-view~ → select "My first view", or evaluate:

#+begin_src emacs-lisp
(org-ql-view "My first view")
#+end_src

Press ~RET~ on any row → message. Press ~g~ → refresh. Press ~q~ → bury.
The muscle memory is already yours.

* The ~actionable-query-defview~ macro

~actionable-query-defview~ registers a named view in ~org-ql-views~.
All keyword arguments are either intercepted by the library (listed
below) or forwarded verbatim to =vtable='s ~make-vtable~.

** Optional symbol name — ~M-x~ command + insert-at-point

Prefix the view title with an unquoted symbol to generate a named
interactive command:

#+begin_src emacs-lisp
;; Old form (still works):
(actionable-query-defview "Gerrit Patches" :objects … :actions '())

;; New form — also defines `my-gerrit' as an interactive command:
(actionable-query-defview my-gerrit "Gerrit Patches" :objects … :actions '())
#+end_src

With the symbol form, ~M-x my-gerrit~ opens the dedicated view buffer
as usual.  The command also accepts an optional argument for
insert-at-point:

| Call                        | Effect                                                 |
|-----------------------------+--------------------------------------------------------|
| ~(my-gerrit)~ / ~M-x my-gerrit~ | Open the view's dedicated buffer (normal behaviour). |
| ~(my-gerrit t)~               | Insert cached content at point in the current buffer. |
| ~(my-gerrit 'fetch-latest)~   | Fire a fresh fetch, then insert at point on delivery. |

This makes it trivial to compose multiple views into a single
"Weekly Review" buffer:

#+begin_src emacs-lisp
(with-current-buffer (get-buffer-create "*Weekly Review*")
  (erase-buffer)
  (insert "* Blocked patches\n\n")
  (my-gerrit 'fetch-latest)
  (insert "\n\n* Reviews needed\n\n")
  (my-reviews 'fetch-latest))
#+end_src

** ~:objects~ — what to show

Three shapes, dispatched at runtime:

#+begin_src emacs-lisp
;; 1. Plain list — static, evaluated once at definition time.
:objects '("apple" "banana" "cherry")

;; 2. 0-arg function — sync thunk, re-called on every open / `g' refresh.
:objects (lambda () (directory-files "~/Downloads" t "\\.pdf\\'"))

;; 3. 1-arg function — async; receives a CALLBACK.
;;    Shows ⏳ Loading… until (funcall callback objects) fires.
;;    `g' re-fires the fetch from scratch.
:objects (lambda (callback)
           (url-retrieve "https://news.ycombinator.com/rss"
             (lambda (_status)
               (funcall callback (aq--parse-rss-items
                                   (xml-parse-region (point) (point-max)))))))
#+end_src

Async callbacks may also deliver *grouped* results — an alternating
=(\"Title\" objects-list …)= plist — and the library renders one titled
vtable per group automatically.

** ~:columns~ — how to render rows

Standard =vtable= column specs; each column is a plist:

#+begin_src emacs-lisp
:columns '((:name "Date"  :width 12
                  :getter (lambda (o &rest _) (plist-get o :date)))
           (:name "Title" :width 60
                  :getter (lambda (o &rest _) (plist-get o :title))))
#+end_src

When omitted, a single fallback column renders each row as ~(format "%s" obj)~.

*** Column alignment

Pass ~:align~ on any column plist.  =vtable= natively supports ~'left~ and
~'right~; actionable-query extends this with ~'center~, which centers both
the header text and cell values within the column width:

#+begin_src emacs-lisp
(actionable-query-defview "Demo: centered columns"
  :objects '("Alpha" "Beta" "Gamma")
  :columns '((:name "Item" :width 20 :align 'center)))
#+end_src

~Item~ headers and cells are centered; omitting ~:align~ defaults to left.

*** Editable cells

Pass ~:editable t~ plus a ~:setter~ on any column plist to let the user
edit that cell in place.  ~e~ prompts with the cell's current value
pre-filled, calls ~:setter~ with the row object and the new string, and
reverts the table so every row gets a chance to repaint — not just the
edited one (see below for why that matters).  The cell under point
gets a distinct face (~aq-editable-cell-face~) the moment point lands
on an editable column, so it is visually obvious which cells respond
to ~e~ before pressing anything:

Editing always goes through the minibuffer prompt — there is no
inline, type-directly-into-the-buffer mode.  Cell text doubles as a
vtable column, and columns carry their own keybindings (sorting via
~S~, narrowing/widening, click-to-sort on the header, ~=~ for column
filtering).  Making cell text itself locally writable would mean
suspending or shadowing all of that, just for the one column being
edited — not worth it for a feature whose job is reading a new value,
not replacing vtable's own column interactions.

#+begin_src emacs-lisp
(actionable-query-defview igo "Temperature Converter"
  ;; Each row keeps a pointer to its sibling, so the `:setter' below can
  ;; reach "the other one" without needing the whole objects list.  Built
  ;; inside `:objects' itself --- not in an outer `let' --- so this works
  ;; under dynamic binding too: `actionable-query-defview' defers opening
  ;; the buffer, and a `let'-bound `celsius'/`fahrenheit' referenced from
  ;; outside this form would already be unbound by the time it runs.
  :objects (let* ((celsius    (list :label "Celsius"    :value "31"   :unit "°C"))
                  (fahrenheit (list :label "Fahrenheit" :value "87.8" :unit "°F")))
             (plist-put celsius    :sibling fahrenheit)
             (plist-put fahrenheit :sibling celsius)
             (list celsius fahrenheit))
  :columns
  '((:name "" :width 12 :getter (lambda (o &rest _) (plist-get o :label)))
    (:name "Value" :width 8 :align 'right
           :editable t
           :getter (lambda (o &rest _) (plist-get o :value))
           :setter (lambda (o new-value)
                     (unless (string-match-p "\\`-?[0-9]+\\.?[0-9]*\\'" new-value)
                       (user-error "Temperature must be a number, got %S" new-value))
                     (let ((n       (string-to-number new-value))
                           (sibling (plist-get o :sibling)))
                       (plist-put o :value new-value)
                       (plist-put sibling :value
                                  (format "%d"
                                          (round
                                           (if (string= (plist-get o :label) "Celsius")
                                               (+ (* n 9.0 (/ 1.0 5)) 32)
                                             (* (- n 32) 5.0 (/ 1.0 9)))))))))
    (:name "" :width 4 :getter (lambda (o &rest _) (plist-get o :unit))))
  :prose-bottom (insert (propertize "Hot! Stay hydrated!" 'face 'success)))
#+end_src

Editing Celsius's ~Value~ to ~100~ auto-updates Fahrenheit to ~212~, and
vice versa — the ~:setter~ mutates both the edited row and its sibling
in one pass.  This is also why ~e~ reverts the whole table rather than
calling ~vtable-update-object~ on just the edited row: a ~:setter~ free
to touch sibling rows needs every row repainted, not only the one
under point.

Columns without ~:editable~ are untouched — pressing ~e~ on the Label
column above signals a clear ~user-error~ rather than silently no-oping.
~:getter~ stays read-only as before; ~:setter~ is the only new surface,
so the column spec doubles as the inverse-function contract: ~vtable~
has no way to infer how to write a value back into an arbitrary object,
so the call site supplies it explicitly.

*** Reacting to the new value: updating ~:prose-bottom~ on edit

A ~:setter~ can do more than write the cell back — it can react to
the new value and update other parts of the buffer.  Here, editing
either temperature also picks a weather message for the Celsius
reading and rewrites the prose line below the table.

This is the point of /actionable/ queries: the table does not just
display a fact, it tells the user what to do about it — "Cold, wear
a jacket" rather than a bare number left for the user to interpret
themselves.  A row that only shows data is a report; a row that also
hands back a reaction is what this package is named after:

#+begin_src emacs-lisp
(defun my/weather-message (celsius)
  (cond
   ((< celsius 0)  "Behold, the frost descends upon the land; clothe thyself well.")
   ((< celsius 10) "A chill prevails; let a jacket be thy companion.")
   ((< celsius 20) "Mild and temperate is this hour; go forth in peace.")
   ((< celsius 30) "The sun smiles gently upon thee; rejoice in this fair day.")
   (t              "Great heat besets the land; drink water, and tarry not in the sun.")))

(actionable-query-defview "Temperature Converter"
  :objects (let* ((celsius    (list :label "Celsius"    :value "31"   :unit "°C"))
                  (fahrenheit (list :label "Fahrenheit" :value "87.8" :unit "°F")))
             (plist-put celsius    :sibling fahrenheit)
             (plist-put fahrenheit :sibling celsius)
             (list celsius fahrenheit))
  :columns
  `((:name "" :width 12 :getter (lambda (o &rest _) (plist-get o :label)))
    (:name "Value" :width 8 :align 'right
           :editable t
           :getter (lambda (o &rest _) (plist-get o :value))
           :setter (lambda (o new-value)
                     (unless (string-match-p "\\`-?[0-9]+\\.?[0-9]*\\'" new-value)
                       (user-error "Temperature must be a number, got %S" new-value))
                     (let* ((n       (string-to-number new-value))
                            (sibling (plist-get o :sibling))
                            (celsius (if (string= (plist-get o :label) "Celsius")
                                         n
                                       (* (- n 32) 5.0 (/ 1.0 9)))))
                       (plist-put o :value new-value)
                       (plist-put sibling :value
                                  (format "%d"
                                          (round
                                           (if (string= (plist-get o :label) "Celsius")
                                               (+ (* n 9.0 (/ 1.0 5)) 32)
                                             celsius))))
                       ;; Found via its `my-weather-message' text property,
                       ;; not by searching for the old wording, so this
                       ;; works regardless of what the message used to say.
                       ;; `save-excursion' matters: without it, point ends
                       ;; up parked in the prose region, and the
                       ;; `vtable-revert' that runs right after this
                       ;; setter returns can no longer find the table.
                       (save-excursion
                         (when-let ((match (text-property-search-forward
                                            'my-weather-message nil nil)))
                           (let ((inhibit-read-only t))
                             (delete-region (prop-match-beginning match)
                                            (prop-match-end match))
                             (goto-char (prop-match-beginning match))
                             (insert (propertize (my/weather-message celsius)
                                                  'face 'success
                                                  'my-weather-message t))))))))
    (:name "" :width 4 :getter (lambda (o &rest _) (plist-get o :unit))))
  :prose-bottom (insert (propertize (my/weather-message 31)
                                     'face 'success
                                     'my-weather-message t)))
#+end_src

Editing Celsius to ~-5~ swaps the message to "Behold, the frost
descends upon the land; clothe thyself well."; edit it to ~25~ and it
becomes "The sun smiles gently upon thee; rejoice in this fair day."
— both directions, since the setter recovers the Celsius reading
whichever side (Celsius or Fahrenheit) was actually edited.  ~vtable-revert~ only touches the
table's own region (~vtable-beginning-of-table~ to
~vtable-end-of-table~), so rewriting the trailing prose text from
inside the setter is safe — it never collides with the table redraw
that ~e~ triggers afterward.

*** A self-contained one-liner: ~Converter: <num> C = <num> F~

Since I work with American colleagues, this is a useful widget for me
to make as a simple one-liner in the form ~Converter: <num> C = <num>
F~.  No header row, no row-per-temperature — just one line, two
editable numbers, kept in sync:

#+begin_src emacs-lisp
(actionable-query-defview xx "C/F Converter"
  :objects (list (list :celsius "31" :fahrenheit "88"))
  ;; `:use-header-line t' puts the (blank) header in Emacs's actual
  ;; header-line, out of the buffer body --- so the buffer shows
  ;; nothing but this one line of content.
  :use-header-line t
  :columns
  `((:name "" :width 11 :getter (lambda (&rest _) "Converter:"))
    (:name "" :width 4 :align 'right
           :editable t
           :getter (lambda (o &rest _) (plist-get o :celsius))
           :setter (lambda (o new-value)
                     (unless (string-match-p "\\`-?[0-9]+\\.?[0-9]*\\'" new-value)
                       (user-error "Temperature must be a number, got %S" new-value))
                     (plist-put o :celsius new-value)
                     (plist-put o :fahrenheit
                                (format "%d" (round (+ (* (string-to-number new-value) 9.0 (/ 1.0 5)) 32))))))
    (:name "" :width 4 :getter (lambda (&rest _) "°C"))
    (:name "" :width 4 :getter (lambda (&rest _) "="))
    (:name "" :width 5 :align 'right
           :editable t
           :getter (lambda (o &rest _) (plist-get o :fahrenheit))
           :setter (lambda (o new-value)
                     (unless (string-match-p "\\`-?[0-9]+\\.?[0-9]*\\'" new-value)
                       (user-error "Temperature must be a number, got %S" new-value))
                     (plist-put o :fahrenheit new-value)
                     (plist-put o :celsius
                                (format "%d" (round (* (- (string-to-number new-value) 32) 5.0 (/ 1.0 9)))))))
    (:name "" :width 4 :getter (lambda (&rest _)  "°F")))
  ;; `e' is the global editable-cell key, but RET reads more naturally
  ;; for a one-line "form" like this --- `aq--edit-current-cell' reads
  ;; point itself, so rebinding RET to it needs no argument plumbing.
  :actions '(("RET" "Edit number" (lambda (&rest _) (aq--edit-current-cell)))))
#+end_src

Renders as ~Converter:  31 C =  87.8 F~ — one object, four columns
(two literal labels, two editable numbers), no visible header.
Pressing ~RET~ on either number edits it and converts straight into
the other, same validation and same whole-table revert as the two-row
version above; the only difference is shape, not mechanism.

** ~:actions~ — what the user can do

A list of ~("key" "description" fn)~ triples. Each triple is:
- registered as a vtable action (bound to =key= in the buffer), and
- collected into the ~?~ transient popup with its description shown.

#+begin_src emacs-lisp
:actions `(("RET" "Open in browser"  ,(lambda (o) (browse-url (plist-get o :url))))
           ("w"   "Copy URL"         ,(lambda (o) (kill-new (plist-get o :url))))

                                                 (plist-get o :title))

#+end_src

- ~r~ — snooze (dismiss) the row per ~:snooze-period~
- ~w~ — copy the row to the kill-ring

** ~:snooze-period~ — how long dismissed rows stay hidden

One of ~'tomorrow~ (default), ~'next-week~, or ~'forever~. Snooze state

** ~:auto-refresh~ — keep the view fresh automatically

A human-readable interval string:

#+begin_src emacs-lisp

** Slow-fetch guard — skip auto-fetch when a view is expensive

When a fetch completes, its duration is recorded globally (keyed by view
name).  On the *next* visit to that buffer, if the last fetch took longer
than ~actionable-query-auto-fetch-slow-threshold~ seconds (default ~1.0~),
the view opens instantly with the cached data already displayed and a
message in the echo area:

#+begin_example
This view took 12 seconds last time, so not auto-fetched; press `g' to fetch latest.
#+end_example

This keeps expensive views — ones that hit slow APIs or take 10+ seconds —
from blocking every visit.  The data is often fresh enough; you reach for
~g~ when you actually want an update.

Set the threshold to ~nil~ to disable the guard entirely and always
auto-fetch regardless of cost:

#+begin_src emacs-lisp
(setq actionable-query-auto-fetch-slow-threshold nil)
#+end_src

Or raise it for your tolerance:

#+begin_src emacs-lisp
(setq actionable-query-auto-fetch-slow-threshold 5.0)  ; skip only if > 5 s
#+end_src

A ~(lambda (obj) string)~ that assigns each row to a named group.

#+begin_src emacs-lisp

** ~:hearting~ — favourite rows across sessions

** ~:org-upsert~ — associate a row with its Org tree (find *or* create)

One keyword owns the whole row↔org-tree round-trip.  ~:org-upsert~ is a
~(lambda (row) …)~ that returns the live marker of the tree the row belongs to,
*finding* an existing tree or *minting* a new one — an upsert.  On each cursor
move the lambda is called; when it yields a live marker, ~org-marker~ and
~org-hd-marker~ are set on that line — enabling standard ~org-agenda-mode-map~
navigation (~RET~, ~TAB~, ~I~, ~o~) to jump to the heading.

Setting ~:org-upsert~ also activates the full org-agenda command suite from
~agenda.el~ — clock in/out, schedule, deadline, TODO cycling, priority, tags,
effort, archive, and kill subtree — all on the standard keys.

The lambda may return a marker directly, or a **key string** (a content-join
key such as a Jira ID); the string case is resolved for you (see the content-join
below).  The find-or-create workhorse is ~actionable-query-upsert-org-tree~:

#+begin_src emacs-lisp
:org-upsert
  (lambda (o)
    (actionable-query-upsert-org-tree
      :key        (plist-get o :jira)                       ; the join key
      :title-spec (list (format "%s %s" (plist-get o :jira) ; else create this
                                (plist-get o :title))
                        :SCHEDULED "<2026-…>")))
#+end_src

~actionable-query-upsert-org-tree~ first looks for a heading in
~org-default-notes-file~ whose ~:AQ-KEY:~ property equals ~:key~ (or whose text
contains it); if found, that marker is returned and nothing is written.
Otherwise it creates ~* TODO <title>~ from ~:title-spec~ (a STRING, or
~(TITLE :K V … BODY)~ where the ~:K V~ pairs become properties and a trailing
string becomes the body) and stamps ~:AQ-KEY: <key>~ so the next call re-finds
it — even if you later rename the heading.  No duplicates, no drift.

*** The clever case: content-join by a shared key

The Gerrit/Jira views never store an Org id anywhere.  A freshly-fetched Gerrit
change carries a Jira ticket like ~FWD-55732~; ~:org-upsert~ hands that string as
the ~:key~, and core scans ~org-default-notes-file~ for the *first heading whose
text contains it* (or whose ~:AQ-KEY:~ matches) and calls that your tree.  So an
incoming work-item "finds its home" purely because you once wrote
~* TODO FWD-55732 …~ — the ticket number **is** the join key.  Rename or move the
heading freely; the association holds as long as the ID (or the stamped
~:AQ-KEY:~) is there.

*** ~actionable-query-resolve-org-markers~ — batch Org heading lookup

When a view shows objects that correspond to Org headings (Jira tickets, git
branches, project names), resolving the marker for each object is the same
two-pass boilerplate every view duplicates.  ~actionable-query-resolve-org-markers~
runs it once and returns a hash-table you feed straight into ~:org-upsert~:

#+begin_src emacs-lisp
(defun actionable-query-resolve-org-markers (objects key-fn &optional fallback-fn)
  "Return a hash-table mapping each object in OBJECTS to an Org marker.
KEY-FN is called on each object and should return a string (e.g. a Jira
ticket ID) that appears somewhere in a heading in `org-default-notes-file'.
Optional FALLBACK-FN (object table) is called for objects still without a
marker after Pass 1, and may set additional entries directly in TABLE.")
#+end_src

**Pass 1** — for each object whose ~(funcall key-fn obj)~ is a non-nil
string, scan ~org-default-notes-file~ with ~re-search-forward~ for any
heading containing that string; store ~point-marker~ in the table.

**Pass 2** — call ~(funcall fallback-fn obj table)~ for each object still
without a marker, if ~fallback-fn~ is non-nil.

Typical wiring — call it once (e.g. in the async deliver callback), stash
You rarely need to call it by hand: ~:org-upsert~ accepts a key-string and runs
the batch resolution + caching for you.  Just return the join key —

#+begin_src emacs-lisp
(actionable-query-defview "My tickets"
  :objects    (lambda (cb) (my-fetch-async cb))
  :org-upsert (lambda (o) (plist-get o :jira-id)))   ; core content-joins this
#+end_src

— or, to also mint a tree when none is found, return a marker via
~actionable-query-upsert-org-tree~ (see the ~:org-upsert~ section above).
~actionable-query-resolve-org-markers~ remains available for callers who want the
raw ~row → marker~ hash themselves (e.g. a bespoke fallback via ~fallback-fn~).

** Aborting async fetches with ~Q~

Pressing ~Q~ while the ⏳ spinner is running immediately stops the
animation and shows "Fetch aborted — press g to retry."  The in-flight
network request (owned by your ~:objects~ lambda's ~url-retrieve~,
~make-process~, etc.) may still complete in the background, but when it
eventually fires its sentinel the deliver callback silently discards the
result — no spurious render, no error.  Press ~g~ to start a fresh fetch.

* Org-agenda command parity (~agenda.el~)

=agenda.el= is loaded automatically and makes any view with ~:org-upsert~
a drop-in replacement for =org-agenda=.  Every org-agenda command
delegates through a thin wrapper that:

1. Reads ~org-hd-marker~ / ~org-marker~ off the current vtable line.
2. If no marker is found, creates a minimal ~* TODO~ heading in
   ~org-default-notes-file~ (with a ~CREATED~ timestamp) and uses that.
3. Calls the corresponding native Org function in the source buffer
   (wrapped in ~org-with-remote-undo~ so undo works across buffers).
4. Refreshes the vtable display.

** ~:org-ql~ — org-ql queries as first-class views

The ~:org-ql~ preset keyword turns any org-ql sexp into a full
actionable-query view in one line:

#+begin_src emacs-lisp
(actionable-query-defview "My TODOs"
  :org-ql (and (todo "TODO") (scheduled :to today)))

(actionable-query-defview "Urgent items"
  :org-ql (and (todo) (priority "A")))

(actionable-query-defview "This week's done"
  :org-ql (and (done) (closed :from -7)))
#+end_src

The preset expands to:

- ~:objects~ — a 0-arg thunk running ~org-ql-select~ across ~org-agenda-files~
  with the ~element-with-markers~ action (so each row carries live Org markers).
- ~:org-upsert~ — extracts ~:org-hd-marker~ from each element, wiring up all
  org-agenda keys automatically.
- ~:columns~ — sane defaults: TODO keyword, priority, headline, scheduled date, tags.
- ~:actions~ — ~RET~ jumps to the heading in another window.

Because ~:org-upsert~ is set, every org-agenda key (~I~, ~t~, ~C-c C-s~, …) works
out of the box, and the view gains all the actionable-query affordances on
top: hover info, snoozing, bulk actions, column filters, ~?~ transient, etc.

** Auto-creating Org headings on demand

When a non-~:org-ql~ view has ~:org-upsert~ set and the current row maps to no
existing heading, pressing any agenda key (~I~, ~C-c C-s~, ~t~, …)
triggers ~aq-agenda--ensure-marker~:

1. Calls ~actionable-query-resolve-org-markers~ to search
   ~org-default-notes-file~ for an existing heading.
2. If none found, inserts at ~(point-max)~:
   #+begin_example
   * TODO <key-string>
     :PROPERTIES:
     :CREATED:  [2026-05-16 Sat 14:30]
     :END:
   #+end_example
3. Returns a live marker to the new heading — subsequent commands use it
   without prompting again.

** Pomodoro timer (live countdown, deadline-based)

A single-row, no-header view that ticks down once a second, with
~p~/~r~/~s~ standing in for the usual Pause/Reset/Skip buttons.  The
countdown is anchored to an absolute wall-clock deadline rather than
counting down once per timer tick — `run-with-timer' fires late (or
even drops a repeat) whenever Emacs is busy, so decrementing a
counter loses time; recomputing ~remaining~ from ~(deadline - now)~
on every tick keeps the clock accurate regardless of how the timer
itself behaves. Completing a Work session increments the footer
count and flips to a Break (and back), exactly like crossing zero
from pressing ~s~:

#+begin_src emacs-lisp
(defconst aq-pomo-work-seconds 60)
(defconst aq-pomo-break-seconds 60)

(defvar-local aq-pomo--mode 'work)
(defvar-local aq-pomo--remaining aq-pomo-work-seconds)
(defvar-local aq-pomo--deadline nil)   ; float-time; nil while paused
(defvar-local aq-pomo--completed 0)
(defvar-local aq-pomo--timer nil)

(defun aq-pomo--duration (mode)
  (if (eq mode 'work) aq-pomo-work-seconds aq-pomo-break-seconds))

(defun aq-pomo--mmss (seconds)
  (format "%02d:%02d" (/ seconds 60) (% seconds 60)))

(defun aq-pomo--bar (mode remaining)
  (let* ((total (aq-pomo--duration mode))
         (width 40)
         (done  (max 0 (min width (round (* width (/ (float (- total remaining)) total)))))))
    (concat "[" (make-string done ?#) (make-string (- width done) ?-) "]")))

(defun aq-pomo--remaining-until (deadline)
  (max 0 (ceiling (- deadline (float-time)))))

(defun aq-pomo--redraw ()
  "Rebuild the single row from buffer-local state and repaint the table
and the \"Completed pomodoros\" footer.
Goes straight through `vtable-objects' + `vtable-revert' rather than
`aq--refresh-fn' / `vtable-revert-command' --- the static `:objects'
path in `actionable-query-defview' never wires up a vtable
`:objects-function', so `vtable-revert-command's own re-fetch is a
no-op for views like this one; redrawing directly here sidesteps
that gap instead of working around it at the macro level."
  (save-excursion
    (when-let ((table (ignore-errors (vtable-current-table))))
      (setf (vtable-objects table)
            (list (list :mode aq-pomo--mode :remaining aq-pomo--remaining)))
      (vtable--clear-cache table)
      (vtable-revert))
    (goto-char (point-min))
    (when-let ((match (text-property-search-forward 'aq-pomo-completed-footer nil nil)))
      (let ((inhibit-read-only t))
        (delete-region (prop-match-beginning match) (prop-match-end match))
        (goto-char (prop-match-beginning match))
        (insert (propertize (format "Completed pomodoros: %d" aq-pomo--completed)
                            'face 'font-lock-comment-face
                            'aq-pomo-completed-footer t))))))

(defun aq-pomo--tick ()
  "Refresh `aq-pomo--remaining' from the wall clock, then handle a zero crossing."
  (when-let ((buf (get-buffer "*Org QL View:Pomodoro Timer*")))
    (when (buffer-live-p buf)
      (with-current-buffer buf
        (when aq-pomo--deadline
          (setq aq-pomo--remaining (aq-pomo--remaining-until aq-pomo--deadline))
          (when (zerop aq-pomo--remaining)
            (when (eq aq-pomo--mode 'work)
              (setq aq-pomo--completed (1+ aq-pomo--completed)))
            (setq aq-pomo--mode      (if (eq aq-pomo--mode 'work) 'break 'work)
                  aq-pomo--remaining (aq-pomo--duration aq-pomo--mode)
                  aq-pomo--deadline  (+ (float-time) (aq-pomo--duration aq-pomo--mode))))
          (let ((win (get-buffer-window buf t)))
            (if win
                (with-selected-window win (aq-pomo--redraw))
              (aq-pomo--redraw))))))))

Renders as ~Work  01:00  [----------------------------------------]~,

re-anchoring a fresh deadline ~remaining~ seconds out — the same
freeze/resume shape as the buttoned version this mimics, just with
keys instead of clicks.

** CI pipeline dashboard (multi-row, per-row hints, no buttons at all)

Motivated by the CI-dashboard example in
[[https://github.com/d12frosted/vui.el][vui.el]] — same idea (a job
table with live progress gauges, summary counts, and per-row
controls), rebuilt here on top of ~vtable~ instead.

Six jobs, a concurrency limit of two running at once, and a half-second
tick that advances each running job by a random step until it settles
— mostly ~passed~, occasionally ~failed~.  No Start button: the
pipeline kicks off the moment the view opens, via ~:prose~, the same
way the pomodoro timer above auto-starts.  No Reset button either —
~g~ does that job for free now that the engine fix above wires a
0-arg ~:objects~ thunk through as vtable's own ~:objects-function~.

Rather than a Retry button/column, this leans further into
"actionable": ~:help-echo~ tells you what you can *do* about the row
under point, not just what its status is — that is the whole point of
an actionable query, a row that hands back a reaction rather than a
bare fact:

#+begin_src emacs-lisp
(defconst aq-ci-concurrency 2
  "How many jobs may run at the same time.")

(defun aq-ci--initial-jobs ()
  (list (list :id 1 :name "lint"        :status 'queued :progress 0)
        (list :id 2 :name "build"       :status 'queued :progress 0)
        (list :id 3 :name "unit"        :status 'queued :progress 0)
        (list :id 4 :name "integration" :status 'queued :progress 0)
        (list :id 5 :name "package"     :status 'queued :progress 0)
        (list :id 6 :name "deploy"      :status 'queued :progress 0)))

(defvar-local aq-ci--jobs (aq-ci--initial-jobs))
(defvar-local aq-ci--timer nil)

(defun aq-ci--settled-p (job)
  (memq (plist-get job :status) '(passed failed)))

(defun aq-ci--bump (job)
  (if (eq (plist-get job :status) 'running)
      (let ((next (+ (plist-get job :progress) (+ 8 (random 12)))))
        (if (>= next 100)
            (plist-put (plist-put (copy-sequence job) :progress 100)
                       :status (if (< (random 100) 80) 'passed 'failed))
          (plist-put (copy-sequence job) :progress next)))
    job))

(defun aq-ci--fill (jobs)
  (let ((running (cl-count-if (lambda (j) (eq (plist-get j :status) 'running)) jobs)))
    (mapcar (lambda (j)
              (if (and (< running aq-ci-concurrency)
                       (eq (plist-get j :status) 'queued))
                  (progn (cl-incf running)
                         (plist-put (copy-sequence j) :status 'running))
                j))
            jobs)))

(defun aq-ci--status-face (status)
  (pcase status
    ('queued  'shadow)
    ('running 'warning)
    ('passed  'success)
    ('failed  'error)))

(defun aq-ci--gauge (progress face &optional width)
  (let* ((width (or width 16))
         (done  (max 0 (min width (round (* width (/ progress 100.0)))))))
    (propertize (format "[%s%s] %3d%%"
                        (make-string done ?#)
                        (make-string (- width done) ?-)
                        progress)
                'face face)))

(defun aq-ci--redraw ()
  "Repaint the table and the summary-counts header from `aq-ci--jobs'.
Same direct `vtable-objects' + `vtable-revert' approach as the
pomodoro demo --- the static `:objects' path never wires up vtable's
own `:objects-function' re-fetch for views that build it this way.
Explicitly seeks the table's own start before calling
`vtable-current-table' --- this runs from a timer, so point may be
sitting anywhere (even on the summary line above the table, where
`vtable-current-table' returns nil) rather than wherever a live user
last left the cursor."
  (save-excursion
    (goto-char (point-min))
    (when-let* ((match (text-property-search-forward 'vtable nil nil))
                (_     (goto-char (prop-match-beginning match)))
                (table (vtable-current-table)))
      (setf (vtable-objects table) aq-ci--jobs)
      (vtable--clear-cache table)
      (vtable-revert))
    (goto-char (point-min))
    (when-let ((match (text-property-search-forward 'aq-ci-summary nil nil)))
      (let ((inhibit-read-only t))
        (delete-region (prop-match-beginning match) (prop-match-end match))
        (goto-char (prop-match-beginning match))
        (insert (aq-ci--summary-text))))))

(defun aq-ci--summary-text ()
  (let ((count (lambda (status)
                 (cl-count-if (lambda (j) (eq (plist-get j :status) status)) aq-ci--jobs))))
    (propertize
     (format "%s  %s  %s  %s"
             (propertize (format "%d passed"  (funcall count 'passed))  'face 'success)
             (propertize (format "%d failed"  (funcall count 'failed))  'face 'error)
             (propertize (format "%d running" (funcall count 'running)) 'face 'warning)
             (propertize (format "%d queued"  (funcall count 'queued))  'face 'shadow))
     'aq-ci-summary t)))

(defun aq-ci--tick ()
  (when-let ((buf (get-buffer "*Org QL View:CI Pipeline*")))
    (when (buffer-live-p buf)
      (with-current-buffer buf
        (setq aq-ci--jobs (aq-ci--fill (mapcar #'aq-ci--bump aq-ci--jobs)))
        (let ((win (get-buffer-window buf t)))
          (if win (with-selected-window win (aq-ci--redraw)) (aq-ci--redraw)))
        (when (cl-every #'aq-ci--settled-p aq-ci--jobs)
          (when aq-ci--timer (cancel-timer aq-ci--timer) (setq aq-ci--timer nil)))))))

(defun aq-ci-retry (job)
  "Re-queue a failed JOB and restart the clock if it had stopped."
  (interactive)
  (let ((id (plist-get job :id)))
    (setq aq-ci--jobs
          (mapcar (lambda (j)
                    (if (eql (plist-get j :id) id)
                        (plist-put (plist-put (copy-sequence j) :progress 0) :status 'queued)
                      j))
                  aq-ci--jobs)))
  (unless aq-ci--timer
    (setq aq-ci--timer (run-with-timer 0.5 0.5 #'aq-ci--tick)))
  (aq-ci--redraw))

(defun aq-ci-explain-failure (_job)
  "Show a mock failure explanation --- stands in for a real log/trace lookup."
  (interactive)
  (message "%s" (shell-command-to-string "fortune | cowsay")))

(defun aq-ci--help-echo (job)
  "Return the SPC/e hint for JOB's status --- the whole point of an
\"actionable\" query: the row doesn't just report a status, it tells
the user what they can do about it."
  (pcase (plist-get job :status)
    ('failed  (propertize "SPC - retry this failed job   e - see an explanation for the failure"
                          'face '(:foreground "red" :weight bold)))
    ('passed  (propertize "Nothing to do here --- this job is green."
                          'face 'success))
    ('running (format "%s is running --- %d%% done." (plist-get job :name) (plist-get job :progress)))
    ('queued  (format "%s is queued, waiting for a free slot." (plist-get job :name)))))

(actionable-query-defview "CI Pipeline"
  :prose (progn
           (setq aq-ci--jobs (aq-ci--initial-jobs))
           (unless aq-ci--timer
             (setq aq-ci--timer (run-with-timer 0.5 0.5 #'aq-ci--tick)))
           (add-hook 'kill-buffer-hook
                     (lambda () (when aq-ci--timer (cancel-timer aq-ci--timer)))
                     nil :local)
           (insert (aq-ci--summary-text)) (insert "\n\n"))
  :objects (lambda () aq-ci--jobs)
  :use-header-line t
  :columns
  `((:name "Job" :width 12 :getter (lambda (o &rest _) (plist-get o :name)))
    (:name "Status" :width 9
           :getter (lambda (o &rest _) (symbol-name (plist-get o :status)))
           :displayer (lambda (v w _)
                        (propertize (truncate-string-to-width v w)
                                    'face (aq-ci--status-face (intern v)))))
    (:name "Progress" :width 26
           :getter (lambda (o &rest _)
                      (aq-ci--gauge (plist-get o :progress)
                                    (aq-ci--status-face (plist-get o :status))))))
  :actions '(("SPC" "Retry failed job" aq-ci-retry)
             ("e"   "Explain failure"  aq-ci-explain-failure))
  :help-echo #'aq-ci--help-echo)
#+end_src

Moving point onto a failed row's ~Status~/~Progress~ cell shows, in
bold red, "SPC - retry this failed job   e - see an explanation for
the failure" — pressing ~SPC~ re-queues that one job and restarts the
clock if it had already stopped; ~e~ stands in for a real log/trace
lookup with ~fortune | cowsay~.  A passed row's hint is simply
reassurance ("Nothing to do here"); running/queued rows report their
own progress — every row answers "what now?", not just "what is it?".

* Calendar / scheduling helpers (~actionable-query-scheduler~)

The companion file ~actionable-query-scheduler.el~ ships utilities for
scheduling items without double-booking your calendar.  It is loaded
automatically by the main file.

** ~actionable-query-next-free-slot~ — find the next open time slot

#+begin_src emacs-lisp
(actionable-query-next-free-slot &optional duration-mins)
#+end_src

Returns an Org timestamp string (e.g. ~<2026-05-16 Sat 14:00>~) for the
first free slot of ~duration-mins~ minutes (default 60).  The search:

1. Inspects today's Org agenda files for all scheduled items that carry an
   explicit ~HH:MM~ time component.
2. Computes each item's end time from its ~Effort~ property (defaulting to 1 h).
3. If an active clock is running /today/, treats ~(clock-start, now)~ as
   busy too — no scheduling over yourself.
4. Finds the first slot (rounded up to the nearest 15-minute boundary) that
   fits within the 08:00–16:00 workday.
5. If today is already full, rolls forward up to 14 days; pops a
   non-blocking dialog announcing the spillover date.
6. Signals ~user-error~ if no slot fits within the 14-day horizon.

#+begin_src emacs-lisp
;; Next 60-minute slot
(actionable-query-next-free-slot)
;; → "<2026-05-16 Sat 14:15>"

;; Next 30-minute slot
(actionable-query-next-free-slot 30)
;; → "<2026-05-16 Sat 09:30>"
#+end_src

** Colorful pill buttons in ~:prose~

=actionable-query= doesn't ship a button renderer, but it takes only a
small self-contained helper to get styled, clickable pill buttons —
the kind you'd put in a ~:prose~ block above a table.

#+begin_src emacs-lisp
(defun my/insert-pill (label action &rest face-overrides)
  "Insert a styled clickable pill button at point.
LABEL is the visible text.  ACTION is either a URL string (opened
with `browse-url') or a nullary function (called directly).
FACE-OVERRIDES is an optional plist of face attributes that replace
the defaults (:foreground \"cyan\", bold italic, released-button box)."
  (let* ((fn         (if (stringp action)
                         (lambda (_) (browse-url action))
                       (lambda (_) (funcall action))))
         (help-echo  (cond ((stringp action) action)
                           ((and (symbolp action) (documentation action))
                            (car (split-string (documentation action) "\n")))
                           (t "Click to run")))
         (face       (append face-overrides
                             '(:foreground "cyan" :weight bold :slant italic
                               :box (:line-width 2 :style released-button)))))
    (insert " ")
    (insert-text-button label
                        'action     fn
                        'follow-link t
                        'face       face
                        'mouse-face 'highlight
                        'help-echo  help-echo)))
#+end_src

#+end_src

* Design notes

** Why vtable?

=vtable= (Emacs 29+) gives us sortable, resizable, filterable columns
for free. Compared to rolling our own tabulated-list-mode, we get:
- ~S~ to toggle sort by column
- ~{~ / ~}~ to narrow/widen columns
- ~M-←~ / ~M-→~ to navigate between columns
- Proper column displayers, formatters, and getters

We add on top: snoozing, hearting, bulk marking, column filtering by
regex, row reordering, auto-refresh, grouped multi-table rendering, and
a ~?~ transient that surfaces every action with its description.

** Why org-ql-views?

=org-ql= already has the infrastructure for named, openable, refreshable
buffer views. We register into its =org-ql-views= alist so ~M-x
org-ql-view~ surfaces all views by name. The =org-agenda-mode= substrate
means ~q~ buries and standard agenda keybindings work without extra
wiring.

** Async model

Emacs is single-threaded. ~url-retrieve~ (without =-synchronously=) is
callback-driven — it starts the request and returns immediately, calling
your function when the response lands. The 1-arg ~:objects~ lambda
receives a CALLBACK; call ~(funcall callback objects)~ whenever your
data is ready. The library seeds the buffer with a ⏳ spinner, swaps in
the real rows on callback, and handles grouped vs flat delivery
transparently.

Pressing ~Q~ during a fetch sets a buffer-local abort flag; the deliver
callback checks it on arrival and silently no-ops.  The underlying
network request finishes in the background and is GC'd when done.

** Snooze persistence

Dismissed rows are stored in ~aq--dismissed~ (a hash table keyed by
~(view-name . expiry-date-or-"forever")~) and added to
=savehist-additional-variables= — so snooze state survives restarts.
Expired daily-snooze keys are pruned automatically on each view open.
Press ~R~ to resurrect all snoozed rows for the current view.

Why snooze?

Whatsapp has no way to snooze messages: To make them disappear for some time!

Another example: 

Jira tickets and Org trees have statuses, but Gerrit patches don't have a
status, however sometimes some work needs to be paused since it's been
deprioritised. There are two options:
1. Perform a cross-check of Gerrit patches against their Jira tickets
   and only surface patches whose Jiras are active.
   - Downside: Some patches have no associated Jira; even worse, these
     two async fetches result in a slow workflow.
2. Make use of Gerrit tags, to tag items as paused then omit them from
   the Gerrit query.
   - Downside: When there's a stack of 10+ items, this can get annoying
     and then I have to remember to remove them later on.

   As such, I can simply snooze them and I'll be reminded something is snoozed
   and can decide to act on that already in-flight work.  

** Org-agenda command parity

~agenda.el~ provides a thin dispatch layer:

- *Every mutating command* (~org-clock-in~, ~org-schedule~, ~org-todo~, …)
  is wrapped in ~org-with-remote-undo~ — so ~C-/~ undoes the source-buffer
  change even from the view buffer.
- *No marker = auto-create*: ~aq-agenda--ensure-marker~ creates a
  ~* TODO~ heading in ~org-default-notes-file~ the first time any agenda
  command is pressed on a row with no linked heading.
- *Display*: ~aq-agenda-show-new-time~ right-aligns a ~S => <date>~ stamp
  on the current line after scheduling, mirroring ~org-agenda-show-new-time~
  exactly (text property, not a real overlay — cleaned up automatically on
  the next vtable revert).
